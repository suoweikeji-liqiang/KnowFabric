一、技术栈总建议
1. 后端主语言：Python
我建议后端主语言就用 Python 3.11/3.12。
原因不是“Python 最先进”，而是它对你这个项目最合适：


文档解析、OCR、多模态调用、数据清洗、抽取流水线，Python 生态最顺


后面你本地 Qwen3.5、多模态页处理、embedding、抽取脚本，基本都更自然


Codex 写 Python 这类工程骨架、管道代码、测试和脚本，成功率通常比较高


你后续要做样本导出、评测、知识包生成，也更方便


一句话：这项目更像“知识工程平台”，不是高并发交易系统，Python 是合适的。

2. Web API 框架：FastAPI
API 层我建议用 FastAPI。
原因很简单：你这个系统天然需要很多结构化接口，而且 FastAPI 基于 OpenAPI/JSON Schema，自动生成交互式 API 文档，这对你后面做外部服务、给别的系统接入非常合适。FastAPI 官方文档也明确强调了基于 OpenAPI、JSON Schema、自动文档和标准 Python 类型声明这些特点。FastAPI+1
我建议它负责：


文档查询接口


chunk 检索接口


fact 查询接口


审核接口


导出接口


任务状态接口


为什么不是 Django
不是不能用，而是 Django 对你这个项目有点“重在管理后台，轻在数据管道”。
你这套系统的核心其实是：


pipeline


indexing


extraction


retrieval


API service


所以我更建议 FastAPI 做主框架，后台页面另做轻前端，而不是一开始把重心放到 Django Admin。

3. 关系型数据库：PostgreSQL
主数据库我建议直接上 PostgreSQL。
原因：


你的主数据有明显结构关系：document、page、chunk、fact、review、job、domain_package


你需要事务、审计、版本链、过滤查询、JSON 字段


后面做结构化事实和轻图谱候选，也很适合放在 Postgres 里先管住


未来要对外服务时，Postgres 作为主真相源也更稳


不要一开始把所有东西都扔到 NoSQL 里，那样前期很快，后面会乱。

4. 向量检索：pgvector
我建议一期直接用 PostgreSQL + pgvector，不要一开始就单独上专门向量数据库。
pgvector 官方项目说明，它支持把向量和业务数据一起存在 Postgres 里，支持精确和近似最近邻搜索，以及 L2、inner product、cosine 等距离；并且能利用 Postgres 的 ACID、JOIN 等能力。它当前支持 Postgres 13+。GitHub
这对你一期特别合适，因为你最重要的不是“亿级向量性能”，而是：


检索结果和主数据放在一起


追溯方便


开发复杂度低


一期足够稳


为什么现在不建议单独上 Milvus / Qdrant / Weaviate
不是说不好，而是你现在的一期重点不是“超大规模向量平台”，而是：


文档资产化


追溯链


事实抽取


领域包


对外服务


如果现在就把数据库、向量库、对象存储、图数据库、搜索引擎全分开，复杂度会明显上升。
一期优先少而稳。

5. 任务队列：Celery + Redis
离线任务层我建议用 Celery + Redis。
Celery 官方当前稳定版文档显示 5.6 为当前稳定版本；官方也明确支持用 Redis 作为 broker，并给出了 celery[redis] 和 redis://localhost:6379/0 这种标准配置方式。Celery 5.6.2 文档+1
这套组合很适合你做：


文档导入任务


PDF 拆页任务


OCR / 多模态任务


embedding 任务


fact 抽取任务


导出任务


局部重跑任务


为什么不是一开始就上 Kafka / Airflow
因为你现在需要的是：


本地单机可跑


任务状态清楚


容易被 Codex 实现


离线批处理稳定


Celery + Redis 在这个阶段更合适。
Airflow 更适合重调度平台；Kafka 更像大规模事件流基础设施，现在都偏重。

6. 文件存储：本地文件系统优先，预留对象存储接口
你现在是本地 3090 服务器，我建议一期：


原始文档


page image


中间 JSON


导出包


都先放 本地文件系统目录。
然后在代码里抽象一个 StorageAdapter，后面要迁到：


MinIO


S3


OSS


NAS


都可以切。
为什么先不强制上 MinIO
如果你现在就上 MinIO 也可以，但我更建议：


一期先让路径简单


开发和调试更轻


仓库初始化更快


如果你非常在意未来对象存储兼容，可以：


本地 dev 用文件系统


预留 MinIO adapter


staging/prod 再换



7. 全文检索：先用 PostgreSQL，后面再评估 Elasticsearch / OpenSearch
一期我建议先别上 ES/OpenSearch。
原因：


你现在更需要先跑通 chunk 主链


pgvector + Postgres 文本检索先足够用


先把“文档 → page → chunk → fact → 回链”打通更重要


等后面出现这种情况时再考虑加 ES：


文档量和 chunk 量大幅上升


复杂过滤和关键词搜索明显变慢


需要更强的高亮、同义词、分词能力



8. 前端：React + Next.js 或 React + Vite
管理台我建议走 React。
如果你想更快起步，我更偏向：


Vite + React + TypeScript：更轻，适合内部后台


如果你很想后期也服务化成产品，再考虑 Next.js


一期页面优先级
一期前端不要做太重，只需要：


文档列表

chunk 检索页


fact 审核页

重点不是“高级 UI”，而是先把数据链条看得见、改得动。

9. ORM / 数据访问层：SQLAlchemy 2.x + Alembic
这个我建议直接定：


SQLAlchemy 2.x


Alembic 做 migration


原因是：


关系模型比较复杂


迁移必须规范


后面 Codex 写 models / migrations / queries 会比较顺


如果你想更轻一点，也可以用 SQLModel，但我还是更建议 SQLAlchemy 正统路线，后面更稳。

10. 配置与数据校验：Pydantic v2
这个和 FastAPI 很搭，适合：


API 请求/响应 schema


pipeline 配置


domain package schema


task payload schema


你后面有很多 JSON 配置、领域包 manifest，这个组合会很舒服。

本地可运行

为优先。
