下面直接给你一版可交给 Codex 的总约束文档初稿。
这份东西的作用，不是描述业务功能，而是给整个项目“立轨道”，避免后面越做越散。

工业与能源领域知识注入平台
开发约束与工程规范 v1
1. 文档目的
本文档用于约束“工业与能源领域知识注入平台”的架构设计、数据设计、文档处理流程、知识抽取流程、接口输出方式、工程组织方式与验收标准。
本文档优先级高于一般实现细节说明。
Codex 在开发过程中，如遇到局部实现方案与本规范冲突，应优先遵循本规范。

2. 项目定位约束
2.1 系统定位
本系统不是普通资料管理工具，也不是单一问答机器人。
本系统定位为：
面向工业设备与能源场景的知识注入底座
首期落地领域为：


暖通领域


执行层/变频器领域


后续预留扩展到：


储能领域


光伏领域


综合能源领域



2.2 首期目标
首期重点实现以下能力：


原始文档接入与统一纳管


文档解析与中间数据生成


检索知识库构建


结构化事实抽取


基础追溯链构建


基础人工审核能力


基础对外服务接口



2.3 首期非目标
首期明确不做或不重点做以下内容：


不做全自动高精度知识图谱推理闭环


不做全自动大规模微调训练平台


不做复杂多租户商业 SaaS


不做直接控制现场设备的在线闭环执行


不做一次性覆盖暖通、储能、光伏全领域


不追求全资料一次性深度解析完成



3. 架构约束
3.1 架构原则
系统必须遵循以下原则：
3.1.1 分层架构
必须采用分层架构，不允许把文档处理、知识抽取、检索服务、审核服务、对外接口混写成单体逻辑。
3.1.2 通用底座 + 领域包
底层能力必须尽量领域无关。
领域特有逻辑应通过领域包、配置模板、标签体系、抽取模板扩展，不允许把暖通专有字段硬编码为全局唯一模型。
3.1.3 离线构建 + 在线查询
文档处理、抽取、向量化等高耗时任务必须以离线任务方式为主。
在线请求以检索、查询、轻量组装为主。
3.1.4 可重建
索引、向量、事实抽取结果、派生知识包都必须可重建。
原始文档与关键中间数据必须持久保存。
3.1.5 增量优先
系统必须支持增量导入、局部重跑、断点续跑，不允许设计为每次全量重建。

3.2 模块边界约束
首期工程建议至少拆分为以下模块：
模块 A：文档接入模块
负责文件扫描、导入、去重、基础标签
模块 B：文档解析模块
负责页面拆分、文本提取、页面图生成、基础结构识别
模块 C：中间数据工厂模块
负责 page/chunk/fact 等中间数据生成
模块 D：检索与索引模块
负责全文索引、向量索引、过滤检索
模块 E：知识抽取模块
负责实体、关系、事实候选生成
模块 F：审核修订模块
负责人工审核、纠错、版本回写
模块 G：对外服务模块
负责 API 输出和对外查询接口
模块 H：领域包模块
负责暖通域、变频器域等领域配置与扩展

4. 数据分层约束
系统必须采用六层数据模型，不允许直接将原始文档跳过中间层直接变成问答数据。
4.1 原始文档层
存储原始文件，是全系统根数据源。
4.2 页面层
按页拆分文档，保存页文本、页图、页摘要、页级标记。
4.3 块级语义层
将页面切分为可检索、可抽取的块级知识单元。
4.4 事实层
从块级语义中抽取结构化事实，用于结构化查询、图谱与样本导出。
4.5 索引层
保存全文索引、向量索引、过滤索引，仅作为查询加速层。
4.6 派生资产层
保存知识包、图谱快照、微调样本、FAQ 样本等派生数据。

4.7 真相源约束
必须明确以下真相源关系：


原始文档层是最底层真相源


页面层与块层是追溯和检索主真相源


事实层是结构化知识真相源


索引层不是主真相源


派生资产层不是主真相源



5. 追溯规范
5.1 基本要求
任意一个知识结果、检索结果、事实结果、图谱关系、微调样本，都必须能够回链到原始证据。

5.2 必备追溯字段
以下字段为强制字段：


source_doc_id


source_doc_name


source_page_no


source_chunk_id


evidence_text


confidence_score


review_status


若存在页面区域定位能力，补充：


page_id


bbox_json


text_offset_start


text_offset_end


若存在模型生成结果，补充：


extraction_method


extraction_model


extraction_version



5.3 追溯链层级
系统必须支持以下四级追溯：


结果 → fact/chunk


fact/chunk → page


page → raw document


document/page/chunk/fact → version lineage



5.4 版本链规范
至少记录以下版本：


doc_version


parse_version


extraction_version


embedding_version


multimodal_version


任何结果变更都应尽量可解释为：


原文变了


解析规则变了


抽取规则变了


模型版本变了



6. 命名规范
6.1 总体要求
命名必须统一、可扩展、可跨领域复用。
禁止同义混用、临时缩写泛滥、字段语义不清。

6.2 表命名规范
数据库表名采用：


小写


下划线分隔


单数或复数风格保持全局一致


建议首期统一使用单数资源名或业务实体名风格，例如：


document


document_page


content_chunk


extracted_fact


review_record



6.3 字段命名规范
字段采用：


snake_case


语义明确


禁止无意义短命名


示例：


page_no


chunk_type


confidence_score


review_status


source_doc_id


禁止：


pno


ctype


conf


stat2



6.4 API 命名规范
API 路径采用：


资源导向


名词为主


版本前缀明确


示例：


/api/v1/documents


/api/v1/chunks/search


/api/v1/facts/query


/api/v1/exports/finetune-samples



6.5 领域命名规范
领域包必须使用稳定标识：


hvac


drive


energy_storage


photovoltaics


不要混用中文目录名、拼音、英文缩写作为系统主标识。

6.6 关系命名规范
事实关系、图谱关系必须采用受控词表，不允许同义关系自由漂移。
例如优先定义：


belongs_to


contains


monitors


controls


indicates


causes


may_cause


applies_to


recommends


sourced_from



7. 文档处理规范
7.1 支持格式
首期支持以下常见格式：


pdf


doc


docx


ppt


pptx


txt


md


png


jpg


jpeg


其他格式统一走扩展接口，不直接耦合到核心流程。

7.2 导入规范
导入时必须完成：


文件扫描


文件 hash 生成


doc_id 分配


路径记录


基础属性记录


去重检测


初始标签生成



7.3 去重规范
至少实现两级去重：
完全重复
按文件 hash 判定
疑似重复
按文件名、页数、标题、相似度等综合判定
完全重复文件原则上不重复深处理。
疑似重复文件应打标，不直接删除。

7.4 解析策略规范
文档解析必须区分：
文本优先解析
适用于可直接抽文本的 PDF / Office 文档
OCR / 多模态补充解析
适用于扫描页、图片页、流程图页、复杂表格页
禁止默认所有页面都走高成本多模态流程。

7.5 块切分规范
chunk 切分必须兼顾：


语义完整性


检索有效性


追溯准确性


不能切得过大导致检索噪声大，也不能切得过碎导致语义破坏。
建议切分依据：


标题层级


段落边界


表格边界


列表步骤边界


故障代码段边界


图注边界



7.6 高价值页面优先规则
首期优先识别并强化处理以下页面类型：


故障代码页


参数表页


操作步骤页


维修流程页


控制框图页


接线图页


系统原理图页


培训课件高信息密度页



8. 知识抽取规范
8.1 抽取原则
抽取必须围绕“业务可用性”展开，不做无边界抽取。

8.2 首期重点抽取对象
设备类


equipment


subsystem


component


sensor


actuator


controller


故障类


fault_code


symptom


cause


risk


repair_action


check_step


控制类


control_target


control_variable


manipulated_variable


constraint


strategy


applicable_scenario


参数类


parameter_name


parameter_group


parameter_meaning


default_value


setting_range


setting_effect



8.3 抽取结果要求
每条抽取结果必须尽量包含：


normalized subject


relation


normalized object


evidence_text


source_doc_id


source_page_no


source_chunk_id


confidence_score


review_status



8.4 审核要求
以下内容建议默认进入人工审核队列或至少支持抽样审核：


故障码解释


维修步骤


参数作用解释


多义关系抽取


跨领域实体映射


低置信度结果



8.5 可信度分层规范
知识结果必须带 trust_level，建议首期采用：


L1：厂家明确规则


L2：工程经验知识


L3：论文或研究结论


L4：模型推断或候选结果


任何对外输出如涉及 L4，应明确标识。

9. 领域包规范
9.1 领域包目标
领域包用于承载各领域的对象模型、标签体系、抽取规则、问答模板与规则模板。

9.2 每个领域包至少包含以下内容


domain_manifest


label_schema


entity_schema


relation_schema


extraction_template


retrieval_profile


answer_template


export_profile



9.3 首期领域包
首期必须至少具备：
hvac 领域包
包含冷机、水泵、冷却塔、AHU、多联机、传感器、控制策略等
drive 领域包
包含 ABB 等变频器、参数、故障码、控制模式、接线与联动等

9.4 领域隔离要求
领域包之间允许复用底层结构，但不得直接互相污染词表与规则。
跨领域知识关联必须通过显式映射配置实现。

10. 接口规范
10.1 首期接口类型
首期对外接口至少包括：


文档查询接口


检索接口


事实查询接口


导出接口


审核接口



10.2 检索接口返回规范
至少返回：


chunk_id


doc_id


doc_name


page_no


text_excerpt


score


confidence_score


review_status



10.3 事实查询接口返回规范
至少返回：


fact_id


subject


relation


object


trust_level


source_doc_id


source_page_no


source_chunk_id


evidence_text


review_status



10.4 错误处理规范
接口必须返回统一错误结构，包含：


error_code


error_message


error_details


request_id



10.5 分页与过滤规范
所有列表型接口必须支持分页。
检索和事实查询必须支持至少以下过滤条件：


domain


brand


equipment_type


document_type


trust_level


review_status



11. 工程组织规范
11.1 仓库组织原则
代码必须模块化，按职责组织，不允许所有逻辑堆在一个目录。
建议目录结构类似：


apps/


services/


domain_packages/


pipelines/


shared/


scripts/


tests/


docs/



11.2 配置管理规范
配置必须外置，禁止在代码中硬编码环境依赖。
至少区分：


dev


test


prod


并区分：


model config


storage config


pipeline config


domain package config



11.3 日志规范
必须有结构化日志。
关键任务日志至少包含：


job_id


stage


doc_id


elapsed_ms


status


error_message



11.4 任务状态规范
长流程任务必须记录状态，推荐至少支持：


pending


running


success


failed


skipped


retrying


canceled



11.5 幂等性要求
导入、解析、embedding、抽取、导出等任务，应尽量支持幂等执行。
同一文档重复导入不应造成混乱数据。

12. 测试规范
12.1 测试层次
至少包含：
单元测试
验证纯函数、解析规则、字段映射、切分逻辑
集成测试
验证导入→解析→块生成→抽取→索引链路
回归测试
验证规则升级后，不破坏已有结构和接口

12.2 样例集要求
必须建立一小批固定样例文档，用于回归测试。
样例应覆盖：


纯文本 PDF


扫描 PDF


图文混排 PDF


参数表页


故障代码页


培训 PPT


ABB/暖通典型资料



12.3 关键验收测试点
至少验证：


文档能成功导入


文档能生成 page


page 能生成 chunk


chunk 能回链到文档页


facts 能生成且带 evidence


API 返回字段符合规范


增量重跑不会产生脏数据



13. 性能与容量规范
13.1 容量规划约束
基于当前约 20GB 原始资料，系统应按至少 50GB 到 120GB 总资产规模做预估，不得按原始体积简单规划。

13.2 性能原则
优先保证：


可持续处理


可增量处理


可断点续跑


可追溯


而不是单纯追求一次性全量最快。

13.3 资源使用策略
必须采用：


页面缓存


OCR 缓存


embedding 缓存


多模态摘要缓存


抽取结果缓存



13.4 优先级队列
处理队列必须支持优先级。
首期高优先级资料：


ABB 变频器资料


故障代码资料


控制手册


参数手册


暖通故障诊断重点资料



14. 安全与审计规范
14.1 审计要求
对人工修订、样本导出、批量删除、版本切换等关键操作，必须记录：


operator


operation_type


target_id


timestamp


before_value


after_value



14.2 删除规范
原始文档不应轻易物理删除。
优先采用逻辑删除或失效标记。

15. 开发流程规范
15.1 Phase 交付原则
必须按阶段交付，禁止试图一次性交付完整平台。
建议最少分为：
Phase 1
文档导入 + 页面层 + 块层 + 基础检索
Phase 2
事实抽取 + 追溯链 + 审核后台基础
Phase 3
领域包机制 + 对外 API + 导出能力
Phase 4
轻图谱 + 微调样本工厂

15.2 每个 Phase 必须产出


可运行代码


README


配置说明


数据结构说明


测试说明


已知限制说明



16. 验收规范
首期系统验收至少满足以下条件：


可成功导入一批原始文档


可生成稳定的 document/page/chunk 结构


chunk 可被全文检索和向量检索


结构化 fact 可生成且带 evidence


所有检索与事实结果可回链到文档页


支持局部重跑和增量处理


支持基础审核和修订


对外接口返回字段符合追溯规范


首期至少支持 hvac 与 drive 两个领域包



17. 对 Codex 的执行要求
Codex 在实现时必须遵循以下要求：


先搭基础骨架，再逐步补功能


先实现数据分层，再实现智能功能


先保证追溯链，再优化抽取效果


不得跳过中间数据直接拼凑问答系统


不得将暖通专有逻辑硬编码为全局通用逻辑


所有长流程必须可观察、可重试、可恢复


优先写清楚数据结构、接口契约、测试样例


对不确定的领域逻辑，应通过配置或模板抽象，不要臆造固定规则
