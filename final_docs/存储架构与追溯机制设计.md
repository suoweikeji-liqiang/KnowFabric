存储架构与追溯机制设计
1. 设计目标
本系统以原始行业资料为输入，面向暖通、变频器、储能、光伏等领域，构建可扩展的知识注入底座。
存储架构设计需要同时满足以下目标：


支持原始文档长期保存与持续扩充


支持文档解析、中间数据生成、知识抽取、检索与导出


支持知识结果可追溯到原始文档、原始页面、原始片段


支持增量处理、断点续跑、版本重建


支持未来对外提供知识检索、结构化查询、图谱查询、样本导出等服务能力


支持资料规模从当前约 20GB 持续增长到更大规模



2. 总体原则
2.1 多层存储，不单点承载
系统不得只保存一种知识形态。
原始文档进入系统后，需被加工为多层资产，并分别存储。
2.2 原文是根，向量是索引
向量、摘要、三元组、微调样本都属于派生数据。
真正的知识根必须是原始文档与可定位的原始证据。
2.3 所有结果必须可回链
任意一条知识结果，必须能够回链到：


原始文档


原始页


原始块


原始片段或区域


2.4 可重建
索引、向量、事实抽取结果、图谱候选、样本导出结果都必须可重建。
原始文档和关键中间数据必须持久保存。
2.5 增量优先
系统必须支持新增资料、局部重跑、阶段级缓存，不得每次全量重建。

3. 存储分层设计
系统采用六层存储模型。

3.1.2 存储对象

Word


PPT


图片


扫描件

解压后的原始资料文件


3.1.3 核心要求


原文件原则上不改写


为每个文件生成唯一文档 ID


保存文件 hash


保存原始路径与导入来源


支持版本记录

任何块、事实、图谱关系、问答结果、微调样本，最终都必须能回到这一层。

3.2 页面层
3.2.1 作用
将原始文档拆分为页面级对象，作为追溯与多模态处理的关键中间层。
3.2.2 适用对象


PDF 按页拆分

Word 按逻辑页或段落片段处理

3.3.1 作用
将页面切分为适合检索、抽取和问答的知识单元，是 RAG 和知识抽取的核心层。

3.4 事实层
3.4.1 作用
把块级内容进一步加工成结构化知识，用于图谱、规则、结构化查询与微调样本生成。
3.4.2 典型事实示例

3.5 索引层

3.6 派生资产层
3.6.1 作用

3.6.2 典型资产

3.6.3 说明

必须能基于前面几层重新生成。

4. 追溯机制设计
系统必须构建四级追溯链。

4.1 一级追溯：结果到知识单元
任意对外输出结果必须先关联到：


一个或多个 chunk_id
或


一个或多个 fact_id


例如：


检索结果返回 chunk 列表


诊断建议返回证据 fact 列表


图谱关系返回支撑该关系的 fact 列表



4.2 二级追溯：知识单元到页面
每个 chunk 或 fact 必须能追溯到：


doc_id


page_id


page_no


source_chunk_id


evidence_text


如果支持页面定位，最好进一步保存：


bbox_json


region_id


text_offset_start


text_offset_end



4.3 三级追溯：页面到原始文档
每个页面对象必须能追溯到：


原始文件路径


文件 hash


文档版本


导入批次


这样即使文件重命名、迁移目录或更新版本，也能保持稳定关联。

4.4 四级追溯：版本追溯
系统必须记录至少三类版本：
原始版本


文档版本 doc_version


解析版本


parse_version


抽取版本


extraction_version


推荐增加：


embedding_version


multimodal_version


rule_template_version


这样未来出现以下情况时仍可追查：


原文未变，但解析器升级


原文更新，页码变化


抽取规则变更，事实内容更新


多模态模型版本变化导致摘要不同



5. 推荐存储组合
我建议采用“文件系统/对象存储 + 关系型数据库 + 检索引擎/向量库 + 派生文件仓”的组合。

5.1 文件系统或对象存储
用于保存：


原始文档


页面渲染图


提取图片


缩略图


中间产物文件


适合保存大文件和非结构化对象。

5.2 关系型数据库
用于保存：


文档主索引


页面信息


块级语义对象


事实级对象


审核记录


版本记录


任务状态


关系型数据库适合做业务主数据和追溯链存储。

5.3 检索引擎 / 向量库
用于保存：


全文索引


向量索引


检索加速字段


这部分只负责高效检索，不作为主真相源。

5.4 派生文件仓
用于保存：


JSON


JSONL


CSV


Parquet


图谱导出


微调样本导出


适合保存中间数据快照与可交换资产。

7. 20GB 资料的容量与性能预估
7.1 原始资料规模判断
当前约 20GB 原始资料，对本系统属于中等规模，不属于超大规模。
如果以离线构建、增量处理为主，完全可以在单机环境中逐步处理。

7.2 实际落地容量预估
原始 20GB 资料进入系统后，最终占用空间通常会明显增加。
建议按以下量级预估：
原始文档
约 20GB
页面图像与抽取中间文件
约 10GB 到 40GB
块级 JSON、事实级 JSON、日志与缓存
约 5GB 到 20GB
向量索引与全文索引
约 5GB 到 20GB
总体预估
建议按 50GB 到 120GB 做容量规划
如果扫描件和图片页很多，容量会继续上升。

7.3 真正影响性能的因素
20GB 本身不是核心瓶颈，真正影响性能的是：
文档数量
文件越碎，任务调度和索引元数据越多
总页数
页数决定页面渲染、OCR、多模态处理成本
扫描件比例
扫描件越多，OCR 和多模态处理越重
表格/图纸/流程图占比
图文复杂页越多，多模态处理越重
是否全量重跑
如果没有增量机制，性能会迅速恶化

8. 算力要求与性能策略
8.1 对当前 3090 环境的判断
3090 足够支撑当前阶段的知识工程任务，但应采用“离线构建 + 增量处理 + 分批解析”的方式。
适合：


批量文档解析


局部多模态页面处理


本地 embedding


结构化抽取


小规模问答服务


不适合：


大并发在线重解析


全库同时重跑多模态深处理


大规模多人同时在线复杂推理



8.2 推荐处理策略
文本优先
先走文本抽取与规则判断，只有需要时再触发视觉模型
高价值优先
优先处理故障码、参数表、维修流程、控制手册、ABB 等重点专题
增量优先
新增文件单独入库，已有文件按 hash 跳过
断点续跑
每个处理阶段都必须落状态，失败后可从阶段继续
缓存优先
页面渲染、OCR、embedding、多模态摘要必须缓存

9. 处理流水线要求
建议将处理流水线拆为独立阶段，每阶段都有输入、输出和状态记录。
阶段 1：文档接入
输出 document 记录与原始文件存储
阶段 2：页面拆分
输出 document_page 与 page_asset
阶段 3：文本抽取与清洗
输出 raw_text / cleaned_text
阶段 4：块切分
输出 content_chunk
阶段 5：多模态增强
输出 page_visual_summary / table_json / diagram_description
阶段 6：事实抽取
输出 extracted_fact
阶段 7：索引构建
输出全文索引与向量索引
阶段 8：派生资产导出
输出图谱包、微调样本、专题知识包
每阶段必须支持：


状态记录


错误日志


重试


跳过


局部重建

11. 本章节结论
本系统输入的是原始文档，但最终不能只保存原文或只保存向量，而应形成以下六层资产：


原始文档层


页面层


块级语义层


事实层


索引层


派生资产层


所有派生结果都必须携带文档级、页级、块级的追溯信息，并支持版本化重建。
对于当前约 20GB 资料规模，建议按 50GB 到 120GB 的总存储规模预留空间，并采用离线构建、增量处理、阶段缓存、局部重跑的架构，以保证系统性能和可维护性。
